## Writing standards (apply to docs going to external readers)

Direct quotes from Brendan's coaching IM (`boeing-data-projects/evidence/brendan-coaching-*.jpeg`), distilled in `boeing-data-projects/docs/brendan-writing-steering.md`:

- **Don't default to PowerPoint** for a small, technically-competent audience. Write a 1-pager pre-read and send it ahead.
- **Backup slides for diagrams** during Q&A only — not the medium.
- **Agenda \+ intent slide** so people know why the meeting is happening.
- **Informing isn't a purpose.** Every section ties to an action — clarify, surface concerns, identify problems.
- **Close with "I will / you will"** explicit actions on both sides.

Team conventions (not from Brendan, but the team works this way — see `boeing-data-projects/docs/Boeing Data Projects — Document & Slide Templates.md`):

- Plain English in the audience's nouns: workbook, custom SQL data source, OPENQUERY, linked server, stored proc, view definition, ERPLN, ERPMart, SCMODs, Tableau Server, Cognos package, BTP/BDC, Lake House.
- Tag observations DECIDED / OPEN / ASK when the tag helps the reader sort what's being asked.
- Posture: students, not architects.
- Read it aloud before sending.

When in doubt about tone/structure for a customer-facing artifact, invoke the `brendan-writing-steering` skill before drafting.

## Conventions

- Content destined for the Claude.ai "Boeing Data Projects" project goes under `boeing-data-projects/`.
- Focus-keeper session state (CONTRACT.md, LATER.md, SESSIONS.md) goes under `projects/<name>/`, not in `boeing-data-projects/`.
- Reference material that spans projects or doesn't belong to any goes in `notes/`.
- Claude config (agents, skills, settings) stays under `.claude/` at the project level, never at the root.
- No build/test/lint to run.
- Don't commit raw datasets without first uncommenting the data lines in `.gitignore`.
